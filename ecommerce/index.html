01 — Core Concepts

Three ways to recommend

A recommendation system predicts what a user wants next. It learns patterns — either from how users behave, or from what the products look like.

👥
Type 01
Collaborative Filtering

"Users like you also bought X." Finds similar users, recommends what they liked. Doesn't need to know anything about the products — only the interactions between users and items.

Algorithm: ALS (Alternating Least Squares)
🏷️
Type 02
Content-Based Filtering

"This is similar to things you viewed." Looks at product attributes — category, description, price — and recommends similar items. Great for new products with no interaction history yet.

Algorithm: TF-IDF + Cosine Similarity
🔀
Type 03 — What real companies use
Hybrid Filtering

Combines both approaches. Collaborative handles known users well. Content-based handles new products well. Together they cover each other's blind spots. You'll build both, then combine them.

Best of both worlds
Key vocabulary you must know

Know these cold before writing a single line of code.

User-Item Matrix
A table: rows = users, columns = products, cells = ratings. Most cells are empty — this sparsity is what makes the problem hard and interesting.
Cold Start Problem
New users or products have no history, so the algorithm can't personalise. Fix: popularity-based fallbacks for new users, content-based for new items.
Matrix Factorization
Breaks the big sparse matrix into two smaller matrices to find hidden patterns. This is what ALS does — it discovers "latent factors" nobody labelled explicitly.
Latent Factors
Hidden patterns the algorithm discovers — like "this user prefers budget outdoor gear" — even if that's never written anywhere in the data.
Sparsity
Most users only interact with a tiny fraction of all products. On Amazon, 99%+ of the user-item matrix is empty. Algorithms must handle this gracefully.
Implicit vs Explicit Feedback
Explicit = star ratings given. Implicit = clicks, purchases, time on page. Real e-commerce mostly has implicit — more abundant but harder to model.
RMSE
Root Mean Square Error — how far off your predictions are on average. RMSE of 1.2 on a 1–5 scale = off by 1.2 stars. Lower is better.
Precision@K / Recall@K
Ranking metrics. Precision@5 = of your top 5 recs, how many relevant? Recall@5 = of all relevant items, how many did you catch? Both matter.
02 — The Math

Don't panic — PySpark handles it

But understanding what's happening under the hood helps you debug errors, tune your model, and talk confidently about it in any presentation or interview.

The deal: PySpark handles all the matrix math automatically. You just set the parameters. But knowing the intuition behind ALS, TF-IDF, and cosine similarity will help you know what to adjust when results look wrong.
Algorithm 01 — Collaborative Filtering
ALS — Alternating Least Squares

You have a big table of users × products. Most cells are empty. ALS fills in the gaps by finding two smaller matrices whose product approximates the original.

# Big sparse matrix R (users × products) R ≈ U × Pᵀ # ALS alternates between solving: Step 1 → fix P, solve for U Step 2 → fix U, solve for P Repeat until RMSE stops improving

Latent factors represent abstract features like "budget shopper" — the algorithm discovers these with no labels.

Algorithm 02 — Content-Based
Cosine Similarity

After converting product descriptions to vectors with TF-IDF, measure the angle between two vectors. Closer to 1.0 = more similar.

cosine_sim(A, B) = (A · B) / (|A| × |B|) # dot product / product of magnitudes 0.0 = totally different 1.0 = identical

"wireless earbuds bluetooth" and "bluetooth wireless headphones" score high — shared vocabulary makes them similar even if different products.

Feature Extraction — How text becomes numbers
TF-IDF in PySpark

TF-IDF measures how important each word is to a specific product vs all products. Common words like "the" get low scores; unique descriptive words get high scores.

TF = word count in this doc / total words in doc IDF = log(total docs / docs with this word) TF-IDF = TF × IDF # High = unique to this product # Low = appears everywhere

In PySpark, TF-IDF is a 3-stage Pipeline:

from pyspark.ml.feature import ( Tokenizer, HashingTF, IDF ) from pyspark.ml import Pipeline pipeline = Pipeline(stages=[ Tokenizer( inputCol="description", outputCol="tokens"), HashingTF( inputCol="tokens", outputCol="rawFeatures"), IDF( inputCol="rawFeatures", outputCol="features") ]) model = pipeline.fit(items_df)
Evaluation metrics — report all three
RMSE
↓ lower is better
Measures prediction accuracy for ratings. Target <1.5 on a 1–5 scale. RMSE alone is beginner-level.
Precision@K
↑ higher is better
Of your top K recommendations, what fraction are actually relevant to this user?
Recall@K
↑ higher is better
Of all relevant items that exist, how many did your top K successfully catch?
03 — Tech Stack

What you're actually running

Use Databricks Community Edition — not Azure HDInsight. Databricks is genuinely free. HDInsight charges by the minute and has burned many students' free credits.

Azure HDInsight warning: It charges per minute even when idle. Forgetting to delete (not just stop) the cluster after a session = significant credit burn. This is why we recommend Databricks CE.
01
Databricks Community Edition

Your cloud environment. Free notebook workspace running PySpark. Sign up at community.cloud.databricks.com — no credit card needed. Both of you sign up and verify you can create a cluster before Week 1 starts.

02
PySpark (Apache Spark)

The data processing engine. Handles datasets too large for pandas. You'll mainly use pyspark.ml.recommendation.ALS and pyspark.ml.feature. Already installed in Databricks — zero setup.

03
MovieLens 100K (starting dataset)

Start small. Well-documented, structured exactly right: user_id, movie_id, rating, timestamp. Once your full pipeline works, swap in a real e-commerce dataset. Free at grouplens.org/datasets/movielens.

04
Flask API — the portfolio upgrade

Wrap your trained model in a Flask endpoint: GET /recommend?user_id=1001 → JSON recommendations. This separates a notebook project from a real deployable product. Your territory — Flask + Python is your stack.

05
GitHub — shared repo + v1.0.0 release

One shared repo. Main = stable. Feature branches per person. Databricks links directly to GitHub. Tag a v1.0.0 release when done with README, setup steps, and sample API calls.

The ALS model — core code
from pyspark.ml.recommendation import ALS from pyspark.ml.evaluation import RegressionEvaluator # 1. Split data train, test = df.randomSplit([0.8, 0.2]) # 2. Build and train the model als = ALS( userCol="user_id", itemCol="item_id", ratingCol="rating", coldStartStrategy="drop", # critical — handles new users nFactors=10, # start here, tune up if needed maxIter=10 ) model = als.fit(train) predictions = model.transform(test) # 3. Evaluate evaluator = RegressionEvaluator( metricName="rmse", labelCol="rating", predictionCol="prediction" ) rmse = evaluator.evaluate(predictions) print(f"RMSE = {rmse:.4f}") # 4. Generate top-10 recommendations for all users user_recs = model.recommendForAllUsers(10)
04 — 5-Week Roadmap

One phase at a time

Don't rush ahead with messy data. Each phase depends on the previous one. Week 1 always takes longer than expected — that's normal and expected.

W1
Week 1
Data Preparation

Sign up for Databricks both of you · Download MovieLens 100K · Load into Spark DataFrame · Handle nulls and duplicates · Encode string IDs to integers with StringIndexer · Split train/test 80/20 · Explore the data with basic stats before modelling

W2
Week 2
Collaborative Filtering (ALS)

Understand ALS conceptually before coding · Train first ALS model · Generate top-10 recs for test users · Evaluate with RMSE · Tune nFactors (try 10, 20, 50) and maxIter · Document what each parameter does and why you changed it

W3
Week 3 — most projects fail here
Cold Start Problem

Identify new users and new items in your test set · Implement popularity-based fallback for new users (top-N most-rated items) · Use content-based recs for new items · Test that your system returns something useful — not a crash or NaN — for cold users

W4
Week 4
Content-Based Filtering + Hybrid

Build TF-IDF pipeline on product descriptions/categories · Compute cosine similarity between items · Generate item-to-item recommendations · Combine with ALS output for hybrid results · Add Precision@K and Recall@K to your evaluation report

W5
Week 5 — the finish line
Deploy + Visualize + Ship

Save trained model with pickle · Build Flask API with /recommend endpoint · Create dashboard showing recommendations for sample users · Write README documentation · Tag v1.0.0 GitHub release · Inspect actual recommendation outputs manually — are they sensible and diverse?

Reality check: If data prep takes 2 weeks instead of 1 — that's completely normal. Data cleaning always takes longer than you expect. Don't rush to Week 2 with messy data; it causes hard-to-debug errors in every phase after.

Four questions to align on before starting
Q1
Which cloud platform?

Databricks CE is the safer choice — no billing risk. Both of you sign up and verify you can create and run a basic cluster before Week 1 starts. If one of you can't log in, you lose a full week.

Q2
Which dataset to start with?

MovieLens 100K to start — small, structured, well-documented. Switch to Amazon Reviews or Mercari once your pipeline works end-to-end. Don't start big.

Q3
How to share code?

One shared GitHub repo. Main = stable. Each person in their own feature branch. Merge via pull requests. Databricks notebooks link directly to GitHub — set this up in Week 1, not Week 4.

Q4
Weekly check-in rhythm?

30-minute call or voice note every Friday. Sync progress, unblock each other, adjust the plan. Use a shared Google Doc to track ownership. Don't wing the collaboration — it drifts fast without structure.

06 — Common Pitfalls

Mistakes most beginners make

These aren't hypothetical — they're the exact issues that trip up first-time recommendation system projects. Know them now, not the hard way.

01
Integer encoding skipped

PySpark ALS requires user_id and item_id to be integers. String IDs like "user_abc" throw a cryptic error. Fix: use StringIndexer before running ALS. This is the #1 first bug everyone hits.

02
NaN predictions not handled

ALS returns NaN for users or items it hasn't seen during training. Without coldStartStrategy="drop", your RMSE evaluation will return NaN too. Always set this parameter explicitly.

03
Too many latent factors too early

Setting nFactors=50 on a small dataset causes overfitting — great training RMSE, terrible test RMSE. Start with 10. Only increase if test set RMSE is still high.

04
Wrong train/test split strategy

A random 80/20 split can put all of a user's interactions in training, inflating your metrics. Proper: hold out the last interaction per user. For a first project, random is fine — just know the limitation when you write up.

05
Deploying before inspecting outputs

Good RMSE doesn't mean good recommendations. Before Week 5, manually inspect recs for 5–10 users. If everyone gets the same 10 popular items, you have popularity bias — a common failure mode that metrics alone don't catch.

06
Azure cluster left running

If you choose Azure, always delete the cluster (not just stop it) after each session. Left running overnight = significant credit burn. This is the main reason we recommend Databricks CE instead.

Portfolio upgrade checklist

Do these to go from a good project to a great one.

+
Flask API with /recommend endpoint

Makes the project callable. GET /recommend?user_id=1001 returns JSON recommendations. Separates a notebook from a real product — and it's fully in your wheelhouse.

+
Report RMSE and Precision@K together

RMSE alone is beginner-level. Adding ranking metrics shows you understand recommendation quality beyond raw prediction error. About 10 extra lines of code.

+
GitHub v1.0.0 release with tagged README

Shows professional release practices. Include setup instructions, dataset info, sample API calls, and your evaluation results. 30 minutes of work — pays off for months.

+
Document your cold start strategy explicitly

Most student projects ignore this entirely. A section in your README explaining how you handled new users vs new items shows real-world thinking — interviewers notice immediately.
